## 6.8 查询优化连接顺序

### 6.8.1 核心描述

多表 JOIN 时，表的访问顺序直接影响查询性能（差异可达多个数量级）。H2 的 `Optimizer` 实现了三层策略，根据表数量自动选择最合适的算法，确保优化时间不会超过查询执行时间本身。关于 H2 优化器的完整框架详见第8章《查询优化器深度解读》。

**优化问题的本质：**

给定 N 张表的 JOIN，需要找到最优的访问顺序（排列）。全枚举的复杂度是 N!——N=10 时已是 3,628,800 个计划，N=12 时接近 5 亿。显然，对于超过 7 张表的查询，暴力枚举不可行。

**三层优化策略：**

```text
策略选择逻辑（Optimizer.calculateBestPlan()）:

  输入: filters[] (TableFilter 数组，长度 = N)

  1. 如果 N == 1:
     直接计算 plan cost，返回

  2. 如果 N <= MAX_BRUTE_FORCE_FILTERS (7):
     使用暴力枚举 (Brute Force All)
     - 生成 filters 的所有排列 (N! 个)
     - 对每个排列计算 cost
     - 选择 cost 最小的

  3. 如果 N > 7:
     使用混合策略 (Hybrid):
       m = getMaxBruteForceFilters(N)  // 暴力枚举前 m 个表
                                        // 约束: P(N,m) * C(N-m,2) < 2000
       // 前 m 个位置用暴力枚举
       // 剩余 N-m 个位置用贪心算法填充

  4. 与遗传算法并行:
     在 1-3 的同时运行遗传算法 (最多 500 代)
     选择两个结果中 cost 较小的

MAX_BRUTE_FORCE = 2000 // 保证优化时间可控
```
```text

**getMaxBruteForceFilters 计算：**

```
```java
function getMaxBruteForceFilters(N):
    m = 0
    total = N
    j = N
    while true:
        // P(N, m+1) * C(N-(m+1), 2) < MAX_BRUTE_FORCE ?
        nextTotal = total * j
        j--
        if nextTotal * j * (j-1) / 2 < 2000:
            total = nextTotal
            m++
        else:
            break
    return m

  示例计算:
    N=10:
      m=0: total=10, j=9,  10*9*8/2=360 < 2000 → m=1
      m=1: total=90, j=8,  90*8*7/2=2520 > 2000 → 停止
      结果: m=1 (只暴力枚举第一个位置)

    N=8:
      m=0: total=8, j=7,  8*7*6/2=168 < 2000 → m=1
      m=1: total=56, j=6,  56*6*5/2=840 < 2000 → m=2
      m=2: total=336, j=5, 336*5*4/2=3360 > 2000 → 停止
      结果: m=2 (暴力枚举前两个位置)
```

**成本计算（Plan.calculateCost）：**

```text
cost = 1
for each filter in plan:
    item = filter.getBestPlanItem(session, filters, filter)
    cost += cost * item.cost   // 复合乘法累积

解释:
  成本模型基于"行数估计"——每个 JOIN 步骤的输入行数乘以
  当前已有行数，估算出中间结果的大小。
  
  cost 初始为 1，每次迭代：
    cost += cost * item.cost
    → cost *= (1 + item.cost)
  
  因此最终成本 ≈ ∏(1 + cost_i)，反映了多表 JOIN 的
  复合增长效应
```

**连接顺序优化的复杂度增长：**

```text
表数量 (N) 与优化复杂度增长曲线：

  N     N! 个排列    暴力枚举可行?    暴力枚举代价
  ──────────────────────────────────────────────
  1       1           ✔               ~1 μs
  2       2           ✔               ~2 μs
  3       6           ✔               ~6 μs
  4       24          ✔               ~24 μs
  5       120         ✔               ~120 μs
  6       720         ✔               ~720 μs
  7       5040        ✔               ~5 ms
  8       40320       接近上限          ~40 ms
  9       362880      不可行            ~360 ms
  10      3628800     不可行            ~3.6 s
  12      479001600   不可行            ~8 分钟

  H2 的适配策略:
    1-7 张表: 暴力枚举, 7! = 5040 约 5ms
    8-10 张表: 混合策略 (暴力 + 贪心)
    10+ 张表: 贪心 + 遗传算法

  成本估算的准确度影响:
    cost = 1 + cost * item.cost
    成本模型是乘法累积的, 因此中间结果大小的
    估计偏差会指数级放大
  
  偏差示例:
    真实行数 = 1000, 估算行数 = 500
    5 表 JOIN 后: 实际成本 = ∏(1000) = 10^15
                 估算成本 = ∏(500) = 3.1*10^13
                 偏差约 32 倍
    
    虽然绝对值偏差大, 但不同排列的相对排序通常是正确的
    因此优化器仍能选择最优(或接近最优)的计划
```

### 6.8.3 暴力枚举排列

**图 6-71: 暴力枚举排列示意图**

场景: 3 表查询 (t1, t2, t3)，3! = 6 个排列

  FROM t1, t2, t3 WHERE t1.a = t2.b AND t2.c = t3.d

  所有排列及成本计算:

  排列 1: [t1, t2, t3]            排列 2: [t1, t3, t2]
    cost = 1                          cost = 1
    + t1: cost=10 → 1+1*10=11        + t1: cost=10 → 11
    + t2: cost=5  → 11+11*5=66       + t3: cost=20 → 11+11*20=231
    + t3: cost=2  → 66+66*2=198      + t2: cost=5  → 231+231*5=1386
    总 cost: 198                      总 cost: 1386

  排列 3: [t2, t1, t3]            排列 4: [t2, t3, t1]
    cost = 1                          cost = 1
    + t2: cost=10 → 11               + t2: cost=10 → 11
    + t1: cost=2  → 11+11*2=33       + t3: cost=4  → 11+11*4=55
    + t3: cost=2  → 33+33*2=99       + t1: cost=2  → 55+55*2=165
    总 cost: 99                       总 cost: 165

  排列 5: [t3, t1, t2]            排列 6: [t3, t2, t1]
    cost = 1                          cost = 1
    + t3: cost=20 → 21               + t3: cost=20 → 21
    + t1: cost=2  → 21+21*2=63       + t2: cost=4  → 21+21*4=105
    + t2: cost=5  → 63+63*5=378      + t1: cost=2  → 105+105*2=315
    总 cost: 378                      总 cost: 315

  最优计划: 排列 3 [t2, t1, t3] cost=99
            ↓
  执行计划:
    t2 × t1 (通过 t1.a = t2.b) → 中间结果
    × t3 (通过 t2.c = t3.d)    → 最终结果

  排列枚举顺序 (Permutations 生成器):

  start: [t1, t2, t3]
        → [t1, t3, t2]
        → [t2, t1, t3]
        → [t2, t3, t1]
        → [t3, t1, t2]
        → [t3, t2, t1]

  Permutations 使用 next() 方法按字典序生成所有排列
  采用经典算法：从右向左找第一个降序对，交换并反转后缀

**排列枚举的剪枝优化：**

```text
暴力枚举中的隐式剪枝——通过 cost early stop:

  枚举 [t1, t2, t3, t4, ...] 的所有排列:

  当前已知最优 cost = 99 (由排列 [t2, t1, t3] 产生)

  正在枚举 [t3, t2, t1, t4, ...]:
    cost = 1
    + t3: cost=20 → 21
    + t2: cost=4  → 21+21*4=105
    ── 已经超过最优值 99!
    停止继续计算, 跳过剩余的 [t4], [t5], ... 排列

  剪枝效果:
    平均减少 30-50% 的排列需要计算完整成本
    最坏情况下仍需要计算所有排列

  基于成本的排列排序:
    有一种优化是: 先根据单个 filter 的估计成本排序
    将低成本 filter 放在前面 → 更快达到 good cost
    → 更多排列被 early stop 剪枝

  树形枚举过程:
                    []
                  / |  \
                t1  t2  t3  ...
               / \   |   |
              t2  t3 t1  t1 ...
              |   |  |    |
              t3  t2 t3  t2 ...

    每层的分支因子逐渐减小
    第 k 层有 (N-k+1) 个选择
    使用 early stop 可以减少实际探索的分支
```

**贪心算法的局部最优陷阱示例：**

```text
贪心选择每一步取局部最优, 可能错过全局最优：

场景: 4 表 JOIN (t1, t2, t3, t4)

真实成本矩阵 (从表 A JOIN 表 B 的 cost):
         t1    t2    t3    t4
  t1     -     10    100   1000
  t2     10    -     5     100
  t3     100   5     -     10
  t4     1000  100   10    -

贪心过程:
  步骤 1: 选择起始表 → 选择 rows 最少的表 t1 (rows=10)
  步骤 2: t1 后选择 -> 找与 t1 JOIN cost 最低的 → t2 (cost=10)
          plan = [t1, t2], cost = 1+1*10 = 11
  步骤 3: t2 后选择 -> t3 (cost=5) 或 t4 (cost=100)
          选 t3 (cost=5): plan = [t1, t2, t3]
          cost = 11 + 11*5 = 66
  步骤 4: 选 t4: plan = [t1, t2, t3, t4]
          cost = 66 + 66*10 = 726
  
  贪心结果: [t1, t2, t3, t4], cost = 726

真正最优:
  从 t3 开始: [t3, t2, t1, t4]
    t3(5) → t2(5) → t1(10) → t4(1000)
    cost = 1+5=6 → 6+6*5=36 → 36+36*10=396 → 396+396*1000=396396
    等等, 这个成本更高...
  
  检查其他排列:
    [t3, t4, t2, t1]:
      t3(5) → t4(10) → t2(100) → t1(1000)
      cost = 6 → 6+6*10=66 → 66+66*100=6666 → 6666+6666*1000=6.6M
  
  结论: 在简单的成本模型下, 贪心结果可能已经接近最优
       对于复杂的选择性估算, 贪心可能错过更好计划
       此时遗传算法通过随机搜索可能找到更好计划
```

```text
  │         │                │    │         │                   │
  │         ▼                │    │         ▼                   │
  │ 第4步: t3 ──→ t4(cost=10)│    │ 第4步: t4 ──→ t3(cost=10)  │
  │         │                │    │         │                   │

  │ plan: [t1,t2,t3,t4]     │    │ plan: [t2,t1,t4,t3]        │

  如果第3步选 t4 (看似更贵 cost=100), 第4步接 t3 (cost=10)
```

### 6.8.5 遗传算法突变操作

**图 6-73: 遗传算法突变操作**

```java
遗传算法的排列编码和突变操作：

  染色体编码:
    每个计划表示为一个排列（filter 的顺序）:
    [t5, t2, t7, t1, t4, t6, t3, t8, t9, t10]
      ↑    ↑    ↑    ↑    ↑    ↑    ↑    ↑    ↑    ↑
      第1  第2  第3  第4  第5  第6  第7  第8  第9  第10

  适应度函数 = 1 / cost (成本越低，适应度越高)

  突变操作 - Swap Two (两两交换):

    当前排列: [t5, t2, t7, t1, t4, t6, t3, t8, t9, t10]
                        ↑                   ↑
                    随机选择两个位置       (pos=2, pos=6)

    交换后:   [t5, t2, t3, t1, t4, t6, t7, t8, t9, t10]
                        ↑                   ↑
                    t7 和 t3 互换位置

  完全重排 (每 128 代触发):

    if i % 128 == 0:
      // 打乱当前排列，跳出局部最优
      shuffle(plan)
      [t5, t2, t7, t1, t4, t6, t3, t8, t9, t10]
        → [t3, t9, t5, t2, t8, t1, t10, t7, t4, t6]

  遗传算法执行流程:

    // 初始化: 使用贪心结果作为起点
    bestPlan = greedyResult.copy()
    bestCost = calculateCost(bestPlan)
    
    for i in 0..500:
      // 生成新排列
      if i % 128 == 0:
        candidate = shuffle(bestPlan.copy())
      else:
        candidate = swapTwo(bestPlan.copy())
      
      // 计算成本
      cost = calculateCost(candidate)
      
      // 保留更优的
      if cost < bestCost:
        bestPlan = candidate
        bestCost = cost
      
      // 提前终止（canStop）
      if optimizer.canStop():
        break
    
    return bestPlan

  提前终止条件 (canStop):
    如果累积优化时间超过 cost * 100μs，停止优化
    理由：优化时间不应超过查询执行时间
```

**遗传算法的突变探索效率分析：**

```text
交换突变 (swapTwo) 的搜索空间覆盖:

  对于一个 N=10 的排列:
    总排列数: 10! = 3,628,800
    遗传算法探索: 最多 500 代
    覆盖比例: 500/3,628,800 = 0.014%

  但随机搜索的效率远高于纯随机:
    原因是利用了"相邻排列相似性":
    一次交换突变产生的排列与父排列共享约 80% 的顺序
    好的排列通过保留好顺序 + 微调逐步优化

  完全重排 (shuffleAll) 的作用:
    每 128 代触发一次
    目的是跳出局部最优
    新的随机起点可能位于更优的解空间区域

  收敛速度分析:
    初始: 贪心结果 (通常已在 top 20%)
    前 50 代: 快速改进 (交换突变找到更好邻居)
    50-200 代: 缓慢改进 (微调优化)
    200-500 代: 基本收敛 (改进很小)

  canStop() 提前终止:
    如果优化时间已经超过 cost * 100μs
    继续优化可能得不偿失
    提前返回当前最优解
```

**图 6-75: 突变 vs 完全重排的探索效率对比**
```text
两种变异策略的搜索效率：

  swapTwo (交换突变):
  ┌─────────────────────────────────────────────────────────────┐
  │  操作: 随机选择两个位置, 交换它们的元素                       │
  │                                                            │
  │  父代: [t5, t2, t7, t1, t4, t6, t3, t8, t9, t10]         │
  │                     ↑              ↑                        │
  │                   pos=2          pos=6                      │
  │                                                            │
  │  子代: [t5, t2, t3, t1, t4, t6, t7, t8, t9, t10]         │
  │                     ↑              ↑                        │
  │                   t3 和 t7 互换                             │
  │                                                            │
  │  保留顺序比例: ~80% (8/10 位置不变)                         │
  │  探索能力: 局部微调 (在父代附近搜索)                         │
  │  适用: 已经接近最优时进行精细调节                             │
  └─────────────────────────────────────────────────────────────┘

  shuffleAll (完全重排):
  ┌─────────────────────────────────────────────────────────────┐
  │  操作: 随机打乱所有元素的顺序 (每 128 代触发)                 │
  │                                                            │
  │  打乱前: [t5, t2, t7, t1, t4, t6, t3, t8, t9, t10]       │

  │  保留顺序比例: ~0% (完全随机)                                │
  │  探索能力: 全局跳跃 (跳出局部最优)                           │
  │  适用: 陷入局部最优时重新开始探索                             │
  └─────────────────────────────────────────────────────────────┘

  两种突变的配合:

  触发时机             操作          目的
  ──────────────────────────────────────────────────
  每代 (6.1-500)         swapTwo       局部搜索优化
  每 128 代            shuffleAll    跳出局部最优

  如果没有 shuffleAll:
    算法很容易在早期就收敛到局部最优
    因为 swapTwo 只在当前排列附近搜索
    无法探索远离当前解的新区域

  如果没有 swapTwo:
    算法退化为纯随机搜索
    每 128 代尝试一个全新的随机排列
    浪费了"好排列共享良好顺序结构"的信息

  协作效果:
    大多数代: swapTwo 精细调节
    偶尔:     shuffleAll 大范围跳跃
    最优解通常在多个"探索-利用"周期后找到
```

### 6.8.8 优缺点

**优势：**
- 自适应策略，小查询快速，大查询不过度优化
- 遗传算法在大查询上避免了全枚举的指数级成本
- `canStop()` 定时中断，防止优化时间超过查询时间
- 暴力 + 贪心 + 遗传三重保障，覆盖各种查询规模

**局限：**
- 遗传算法停留在局部最优概率较高（只有交换突变，无交叉）
- 成本估算依赖 `getBestPlanItem()` 的估计准确度
- 当表间关联复杂时，贪心阶段可能错过真正最优排列
- 缺乏动态规划中的"最优子结构"保证

### 6.8.9 设计权衡

**为什么不用动态规划（如 System R 算法）：**

System R 的动态规划算法复杂度约 O(3^N)，对 10+ 张表仍然过大。H2 的混合策略是"有限暴力 + 贪心 + 随机搜索"的务实选择。

```text
各算法复杂度对比 (N = 表数):

  算法               复杂度          N=10          N=20
  ──────────────────────────────────────────────────────
  暴力枚举 (Brute)   O(N!)          3.6M          2.4×10^18
  动态规划 (System R) O(3^N)        59049         3.5×10^9
  混合 (H2 Hybrid)   O(P(N,m)*(N-m)²) ~2000       ~4000
  贪心 (Greedy)      O(N²)          100           400
  遗传 (Genetic)     O(generations*N) 5000         10000

  为什么 H2 不采用动态规划:
    动态规划虽然比暴力枚举好 (O(3^N) vs O(N!)),
    但对 10+ 表仍然过大
    嵌入式数据库的内存和 CPU 资源有限
    混合策略在实际中已足够好

  各策略的理论最优性保证:
    暴力枚举:    100% 保证找到最优
    动态规划:    100% 保证找到最优 (System R)
```

**混合策略的阈值设计：**

`MAX_BRUTE_FORCE = 2000` 保证每个查询的优化时间可控。遗传算法的 500 代上限配合 `canStop()` 定时检查，进一步减少优化开销。对于嵌入式数据库，查询不用太长——优化时间超过查询时间就是浪费。

**为何没有交叉操作：**

经典的遗传算法包含交叉（crossover）操作——交换两个染色体的片段。H2 的实现只有交换突变和完全重排，缺少交叉。这可能是出于简化实现和避免破坏已建立的良好连接关系的考虑。完全的排列交叉需要复杂的修复机制（如 PMX、OX 等），H2 选择了更简单的方案。

## 6.9 空间索引 (R-Tree)

> **参考**: H2 官方文档《MVStore》(`h2/src/docsrc/html/mvstore.html#r_tree`)
> 提供了 MVRTreeMap 的完整使用示例代码，包括空间键的添加和相交查询。

### 6.9.1 核心描述

R-Tree 是一种**平衡树**，用于索引多维空间数据（如地理坐标、矩形区域）。扩展类 `MVRTreeMap` 覆盖了 `MVMap` 的 B-Tree 操作，替换为空间感知版本。其核心思想是利用空间邻近性——空间上接近的对象被组织在同一个树节点中，每个节点对应一个最小外包矩形（MBR, Minimum Bounding Rectangle）。R-Tree 在 SQL 查询中的具体应用详见第7章《SQL 执行全流程》。

**空间键（Spatial）：**

```text
默认 2D 矩形，由最小/最大坐标定义：

Spatial (default: 2D rectangle)
  ├── min(0): double    // X 轴最小值
  ├── max(0): double    // X 轴最大值
  ├── min(1): double    // Y 轴最小值
  └── max(1): double    // Y 轴最大值

空间关系判定（SpatialDataType）:
  overlap(a, b):  // AABB 重叠测试
    return a.min(0) <= b.max(0) && a.max(0) >= b.min(0)
        && a.min(1) <= b.max(1) && a.max(1) >= b.min(1)

  contains(a, b):
    return a.min(0) <= b.min(0) && a.max(0) >= b.max(0)
        && a.min(1) <= b.min(1) && a.max(1) >= b.max(1)

  area(spatial):
    return (max(0)-min(0)) * (max(1)-min(1))

  面积增量 (getAreaIncrease):
    // 计算将新键加入现有 MBR 后面积的增加量
    newMBR = union(existingMBR, newKey)
    return area(newMBR) - area(existingMBR)
```

**R-Tree 结构：**

```text

           每个子节点的最小外包矩形

            ┌────────────────────────────┐
            │   Root (Internal Node)     │
            │   MBR[0] = (0,0,5,5)      │
            │   MBR[1] = (6,0,10,10)    │
            │   children[2]             │
            └────────┬────────┬─────────┘

  │ Node A (Internal)│        │ Node B (Leaf)     │
  │ MBR[0]=(0,0,3,5) │        │ objects:          │
  │ MBR[1]=(3,2,5,5) │        │  ObjG=(6,1,8,5)  │

  │ Leaf C  │ │ Leaf D  │
```

**查找（findIntersectingKeys）：**

```text
findIntersectingKeys(x):
    // 深度优先遍历，带有 AABB 重叠测试
    recurse(p, x):
        for each key in p:
            if overlap(p.key[i], x):   // AABB 重叠检测
                if p.isLeaf():
                    yield (key, value)
                else:
                    recurse(p.getChildPage(i), x)

  剪枝效果:
    如果根节点的 MBR 与查询区域不重叠，整棵树都不需要访问
    如果某个内部节点的 MBR 不重叠，该子树完全跳过
    平均每次查询只访问树中 20-40% 的节点
```

**插入（operate）：**

```text
insert(key, value):
    // 1. 从根开始，选择面积增量最小的子树
    p = root.copy()          // COW 副本
    while not p.isLeaf():
        minIncrease = INF
        bestChild = -1
        for each child i:
            increase = getAreaIncrease(p.key[i], key)
            if increase < minIncrease:
                minIncrease = increase
                bestChild = i
        p = p.getChildPage(bestChild)

    // 2. 在叶子插入
    p.insertLeaf(key, value)

    // 3. 向上更新 MBR
    while parent != null:
        updateBounds(parent.key[index])

    // 4. 如果节点太满 → 分裂
    if p too large:
        split(p)

  最小面积增量选择:
    直觉：选择面积增加最少的子树，使 MBR 增长最小
    保持 MBR 紧凑 → 减少重叠 → 提高查询效率
    如果有多个子节点面积增量相同，选择总面积更小的
```

### 6.9.3 线性分裂 vs 二次分裂对比

**图 6-83: 线性分裂 vs 二次分裂对比**

```text
R-Tree 分裂算法的两种策略：

  节点已满（超过 maxEntries），需要分裂为两个节点

  Linear Split (O(N)):

    步骤 1: 选择种子 (Seeds)
      for each dimension d (0, 1):
        // 找到维度 d 上具有最小下界的对象
        lowestMin = argmin(objects[i].min(d))
        // 找到维度 d 上具有最大上界的对象
        highestMax = argmax(objects[i].max(d))
      
      在维度 0 上:
        min(0)=0 (ObjA), max(0)=10 (ObjH)
        → 种子对: (ObjA, ObjH) 在维度 0 上间距最大

    步骤 2: 分配剩余对象
      for each remaining object:
        // 计算分配到种子 A 和种子 B 的面积增量
        increaseA = getAreaIncrease(groupA.mbr, object)
        increaseB = getAreaIncrease(groupB.mbr, object)
        // 分配到增量较小的组
        if increaseA < increaseB:
            groupA.add(object)
        else:
            groupB.add(object)

    效果: 快速但 MBR 重叠可能较大

  Quadratic Split (O(N²)):

    步骤 1: 选择种子 (Seeds) — 二次方选择
      maxWaste = -INF
      for each pair (a, b):
        // 计算"浪费面积"：
        // area(union(a,b)) - area(a) - area(b)
        waste = area(union(a,b)) - area(a) - area(b)
        if waste > maxWaste:
            maxWaste = waste
            seedA = a, seedB = b

      原理: 选择两个相距最远的对象作为种子
      O(N²) 的复杂度确保了种子选择更精确

    步骤 2: 分配剩余对象
      for each remaining object:
        // 同 Linear Split
        increaseA = getAreaIncrease(groupA.mbr, object)
        increaseB = getAreaIncrease(groupB.mbr, object)
        if increaseA < increaseB:
            groupA.add(object)
        else:
            groupB.add(object)

    效果: 分裂质量更高，MBR 重叠更少

  对比示例:

    原始节点 (6 个对象):
      ObjA(0,0), ObjB(1,5), ObjC(3,2), ObjD(4,6), ObjE(7,1), ObjF(8,7)

    Linear Split 结果:

      Group B: ObjB(1,5), ObjE(7,1), ObjF(8,7)  → MBR: (1,1,8,7)
```

        [Root_R]                       [Root_P]
        /       \                      /             NodeR1   NodeR2              NodeP1   NodeP2
      /    \    /    \             /    \    /         L1   L2  L3   L4           P1L1  P1L2 P2L1 P2L2

     Join 算法:

```text
        1. overlap(Root_R, Root_P) → YES → 递归到子节点
        2. overlap(NodeR1, NodeP1) → YES → 继续
        3. overlap(NodeR1, NodeP2) → NO → 剪枝
        4. overlap(L1, P1L1) → YES → 检查所有对象对
        5. overlap(L1, P1L2) → NO → 剪枝

     ┌─────────────────────────────────────────────────────────────┐
     │ 空间连接的剪枝效率取决于 MBR 的紧密程度                       │
     │ MBR 越紧密 → 重叠检测越精确 → 剪枝率越高                      │
     │ 这就是 R-Tree 分裂质量直接影响查询性能的原因                    │
     └─────────────────────────────────────────────────────────────┘
```

### 6.9.7 优缺点

**优势：**
- 范围查询高效（AABB 剪枝大幅减少检查量）
- 树形结构支持平衡的空间分割
- 支持多维度（默认 2D，可扩展）
- 与 MVMap 的 COW 机制无缝集成

**局限：**
- 高维数据"维度诅咒"（>10 维性能急剧下降）
- 分裂算法选择影响树质量（Linear 快但质量差，Quadratic 好但慢）
- 对象形状必须是矩形（MBR 近似，可能产生假阳性）
- 重叠可能导致多个子树都需要访问

### 6.9.8 设计权衡

**Linear vs. Quadratic 分裂：**

| 特性 | Linear Split | Quadratic Split |
|------|-------------|-----------------|
| 复杂度 | O(N) | O(N²) |
| 树质量 | 较低（MBR 重叠较多） | 较高（MBR 重叠较少） |
| 查询性能 | 中等 | 更好 |

```text
分裂策略选择决策:

                    数据重叠程度?
                    │
          ┌─────────┴──────────┐
          │                     │
        低 (大部分分离)        高 (大量重叠)
          │                     │
          ▼                     ▼
  ┌─────────────────┐  ┌─────────────────────┐
  │ Linear Split ✓  │  │ 需要减少重叠?        │
  │                 │  │                      │
  │ O(N) 快速分裂    │  │  YES              NO │
  │ 重叠可接受       │  │   │                │  │
  │ 查询效率足够     │  │   │                │  │
  └─────────────────┘  │   ▼                ▼  │
                        │ ┌────────┐  ┌───────┐│
                        │ │Quadratic│  │Linear ││
                        │ │Split    │  │Split  ││
                        │ └────────┘  └───────┘│
                        └──────────────────────┘

  H2 默认: Linear (O(N))
  可配置 quadraticSplit=true 使用 Quadratic (O(N²))
  建议: 数据高度重叠时选择 Quadratic
```

H2 默认使用 Linear 分裂，可通过 `quadraticSplit=true` 配置切换为 Quadratic。对于大部分地理空间查询，Linear 已经足够，Quadratic 在数据高度重叠时提升明显。

**R-Tree vs. Grid Index：**

网格索引固定网格大小，适合均匀分布数据。R-Tree 自适应数据分布，适合非均匀数据。H2 选择 R-Tree 是因为空间索引的通用性需求。

```text
R-Tree vs. Grid Index 对比:

  特征               R-Tree                  Grid Index
  ──────────────────────────────────────────────────────────
  结构              平衡树                  固定网格
  自适应            是 (数据驱动)            否 (网格大小固定)
  均匀数据          良好                    优秀
  非均匀数据        优秀                    差 (空网格多)
  高维              支持 (维度诅咒)          支持 (网格数爆炸)
  动态插入          ✓                       需重建
  范围查询          高效                    高效
  点查询            O(log N)                O(1) 哈希
  实现复杂度        复杂                    简单

  H2 的选择: R-Tree
  原因:
    · 空间数据通常非均匀分布 (地理上聚集)
    · 支持动态插入和删除 (MVStore 的 COW 模式)
    · 通用性更强, 支持各种空间查询
    · 网格索引不适合 H2 的 append-only 存储模型
```

**COW 适配：** MVRTreeMap 的 `operate()` 比 MVMap 更复杂，因为 R-Tree 的插入位置需要计算面积增量，而且分裂后需要更新父节点的 MBR 边界。COW 模式下，这需要沿路径复制所有受影响的页面。

## 6.10 SQL 解析 (Recursive Descent)

### 6.10.1 核心描述

```text
H2 的 SQL 解析器采用经典的两阶段架构：**Tokenizer（词法分析）→ Parser（递归下降语法分析）**。这种分离使解析器的实现更清晰——Tokenizer 处理字符级细节，Parser 专注于 Token 级别的语法结构。解析器在 SQL 执行全链路中的位置详见第7章《SQL 执行全流程》。
```

**两阶段架构概览：**

```text
SQL 字符串 → Tokenizer → Token 序列 → Parser → AST (语法树)

  "SELECT a.id FROM users WHERE a.name = 'Alice'"
       │
       ▼
  Tokenizer.next():
    → SELECT keyword
    → identifier "a"
    → DOT
    → identifier "id"
    → FROM keyword
    → identifier "users"
    → WHERE keyword
    → identifier "a"
    → DOT
    → identifier "name"
    → EQUAL
    → STRING "Alice"
       │
       ▼
  Parser.parsePrepared():
    → 识别第一个 Token 为 SELECT
    → 调用 parseQuery()
      → 调用 parseSelectExpressions()
      → 调用 parseTableReference()
      → 调用 parseExpression() for WHERE
       │
       ▼
  AST:
    Select
      ├── SelectList: [ColumnExpr(a.id)]
      ├── FromClause: Table(users)
      └── WhereClause: Compare(=, Column(a.name), String(Alice))
```
```text

**Tokenizer 设计：**

Tokenizer 是一个字符级的状态机，读取 SQL 字符串中的下一个 Token。

```
```java
Tokenizer.next():
    skipWhitespace()
    if endOfInput: return null
    
    c = currentChar()
    
    switch true:
        case isIdentifierStart(c):
            return scanIdentifierOrKeyword()
        case isDigit(c):
            return scanNumber()
        case c == '\'':
            return scanString()
        case c == '"':
            return scanQuotedIdentifier()
        case c == '=':
            return EQUAL (0x3C0)
        case c == '<':
            return LESS 或 BIGGER_EQUAL 或 NOT_EQUAL
        case c == '>':
            return BIGGER 或 BIGGER_EQUAL
        case c == '(':
            return OPEN_PAREN
        case c == ')':
            return CLOSE_PAREN
        case c == ',':
            return COMMA
        case c == ';':
            return SEMICOLON
        // ... 约 30 个字符/运算符的分派

  Token 编码:
    Token.java 使用位掩码编码：
      EQUAL       = 0x3C0
      BIGGER      = 0x3E0
      BIGGER_EQUAL = 0x3E1
      LESS        = 0x360
      LESS_EQUAL  = 0x361
      NOT_EQUAL   = 0x362
    这种编码允许使用位运算快速判断 Token 类别
```
```text

**Parser 设计（Recursive Descent）：**

Parser 是一个递归下降解析器，每个语法规则对应一个方法。

```
```java
parsePrepared():                    // 顶级方法：根据 Token 类型分发
    switch (currentTokenType):
        SELECT|VALUES|WITH → parseQuery()
        INSERT → parseInsert() | UPDATE → parseUpdate()
        DELETE → parseDelete() | CREATE → parseCreate()
        ALTER → parseAlter()   | DROP → parseDrop()
        TRUNCATE → parseTruncate() | MERGE → parseMerge()
        CALL → parseCall()     | EXPLAIN → parseExplain()
        // ... 50+ 种语句

parseQuery():                       // SELECT 查询
    read("SELECT"); parseSelectExpressions(); read("FROM")
    parseTableReference()
    if readIf("WHERE"):  parseExpression()
    if readIf("GROUP"):  read("BY"); parseExpressionList()
    if readIf("HAVING"): parseExpression()
    if readIf("ORDER"):  read("BY"); parseOrderList()
    if readIf("LIMIT"):  parseLimit()
    if readIf("OFFSET"): parseOffset()
    return query

// 递归下降表达式解析（每层对应一个优先级）
parseExpression(): parseAnd()                     // OR (最低优先级)
    while readIf("OR"): ...
parseAnd():      parseCondition()                 // AND
    while readIf("AND"): ...
parseCondition(): parseConcat()                   // 比较
    if compareOperator(): return Compare(op, ...)
parseConcat():  parseSum()                        // 连接 ||
    while readIf("||"): ...
parseSum():     parseTerm()                       // + - (左递归→循环)
    while readIf("+"|"-"): ...
parseTerm():    parseFactor()                     // * /
    while readIf("*"|"/"): ...
parseFactor():                                    // 原子表达式（最高优先级）
    if "(":  expr = parseExpression(); read(")")
    if NUMBER|STRING: return ValueExpression.get()
        if token is IDENTIFIER:
            return parseColumnOrFunction()
        ...
```

```text
SELECT 语句的完整递归下降解析调用树：

  parsePrepared()
    │
    ├─ 识别: currentToken == SELECT
    │
    └─ parseQuery()
         │
         ├─ read("SELECT")        // 消费 SELECT 关键字
         │

         │    ├─ parseExpression()     // 第一个列表达式
         │    │    ├─ parseAnd()
         │    │    │    └─ parseCondition()
         │    │    │         └─ parseConcat()
         │    │    │              └─ parseSum()
         │    │    │                   └─ parseTerm()
         │    │    │                        └─ parseFactor()
         │    │    │                             └─ parseColumnOrFunction()
         │    │    │                                  ├─ 识别: a.name
         │    │    │                                  │   → Column("a","name")
         │    │    │                                  └─ 或识别: COUNT(*)
         │    │    │                                      → Function("COUNT")
         │    │    └─ (循环) readIf(",") → 解析下一个表达式
         │    │         └─ parseExpression()
         │    │              └─ ...
         │    └─ (继续直到没有更多逗号)
         │
         ├─ read("FROM")
         │
         ├─ parseTableReference()
         │    ├─ parseTable()     // 基础表名
         │    │    └─ 识别: "users" → Table("users", "a")
         │    │
         │    ├─ (可选) readIf("JOIN")
         │    │    └─ parseJoin()
         │    │         ├─ parseTable()  // JOIN 的表
         │    │         ├─ read("ON")
         │    │         └─ parseExpression()  // JOIN 条件
         │    │              └─ ...
         │    │
         │    └─ (可选) readIf("LEFT"|"RIGHT"|"INNER"|"OUTER")
         │         └─ ...
         │
         ├─ (可选) readIf("WHERE") → parseExpression()
         │    └─ parseAnd()
         │         └─ parseCondition()
         │              └─ parseConcat()
         │                   └─ parseSum()
         │                        └─ parseTerm()
         │                             └─ parseFactor()
         │                                  └─ 识别: o.amount > 100
         │                                      → Compare(>,
         │                                         Column(o.amount),
         │                                         100)
         │
         ├─ (可选) readIf("GROUP") → read("BY") → parseExpressionList()
         │
         ├─ (可选) readIf("HAVING") → parseExpression()
         │
         ├─ (可选) readIf("ORDER") → read("BY") → parseOrderList()
         │
         └─ (可选) readIf("LIMIT") → parseLimit()
              │
              └─ (可选) readIf("OFFSET") → parseOffset()

  递归深度:
    最坏情况: parseExpression → parseAnd → parseCondition → parseConcat
              → parseSum → parseTerm → parseFactor
    最大深度: 7 层 (不包含嵌套子查询)
    嵌套子查询: 递归深度可达到任意深度（但 SQL 通常不超过 10 层）

  左递归处理:
    SQL 语法中的左递归（如 expression ::= expression '+' term）
    通过循环而非递归处理:
      parseSum():
          left = parseTerm()
          while readIf("+"):
              right = parseTerm()
              left = new Plus(left, right)
          return left
    这样避免了无限递归。
```

**运算符优先级和结合性总结:**

```text

─────┼───────────────────┼─────────────────────────┼─────────┼────────
1    | parseFactor       | 括号,列,字面量,函数    | —        | 最高
2    | parseTerm         | *, /, %                 | 左结合   | 高
3    | parseSum          | +, -, &, |              | 左结合   | 中高
4    | parseConcat       | ||                      | 左结合   | 中
5    | parseCondition    | =, <>, <, >, <=, >=     | 左结合   | 中低

7    | parseExpression   | OR                      | 左结合   | 最低

  - AND 优先级高于 OR: a OR b AND c → a OR (b AND c)
  - NOT 优先级高于 AND/OR: NOT a OR b → (NOT a) OR b
  - 相同优先级的运算符左结合: a + b - c → (a + b) - c
  - BETWEEN 包含边界: a BETWEEN x AND y → a >= x AND a <= y
```

### 6.10.5 解析器错误处理流程

**图 6-97: 解析器错误处理流程**

```java
// 两遍解析 + 期望 Token 列表的错误处理
// 第一遍：完整解析
try { parsePrepared(); return statement; }
catch (ParseException e) {
    // 记录错误位置，准备带恢复的第二遍解析
    tokenizer.reset(e.getErrorPos());
}
// 第二遍：带错误恢复的详细错误报告
try { parsePrepared(); }
catch (ParseException e2) {
    expectedTokens = getExpectedTokens(e2.getErrorPos());
    throw new SQLException(
        "Syntax error in SQL statement \"" + sql + "\"",
        errorPosition, expectedTokens);
}

getExpectedTokens(errorPos):
    // 在错误位置收集所有合法 Token
    switch (currentContext):
        AFTER_SELECT:  IDENTIFIER|ASTERISK|OPEN_PAREN
        AFTER_FROM:    IDENTIFIER|OPEN_PAREN
        INSIDE_WHERE:  IDENTIFIER|STRING|NUMBER|OPEN_PAREN|EXISTS|NOT

// 示例：SELECT * FORM users → "Expected: FROM, identifier, Got: FORM"
// 容错：read(tokenType) 抛出异常 vs readIf(tokenType) 返回 false
read(tokenType):    if currentToken != tokenType → throw ParseException
readIf(tokenType):  if currentToken == tokenType → consume & return true
                    else → return false  // 用于可选的语法部分
```

**错误恢复策略总结:**

```text
恢复策略              | 实现方式                                     | 适用场景
──────────────────────┼──────────────────────────────────────────────┼─────────────────────────
read() 抛出异常       | 检查当前 Token, 不匹配则抛出 ParseException  | 必须的语法部分(SELECT等)
readIf() 跳过         | 匹配则消费, 不匹配则直接返回 false           | 可选部分(WHERE,GROUP BY)
readIfWithHint()      | readIf + 预期提示                             | 有多个选项的可选部分
两遍解析              | 第一遍失败后重置并再次解析收集更详细的错误    | 向用户报告友好错误信息
期望 Token 列表       | 在错误位置枚举所有合法的 Token 类型           | 错误消息中的 "Expected: "
错误位置定位          | tokenizer.getTokenStart() 记录精确位置        | 错误消息中的指向箭头

H2 解析器错误处理的核心理念:
  1. 解析错误 = 语法错误(不是系统错误)
  2. 用户需要知道: 哪里错了 + 期望什么 + 实际是什么
  3. 两遍解析: 第一次完全解析, 第二次收集期望 Token 列表
  4. 错误消息尽量友好: 给出上下文 + 期望列表 + 错误位置
  5. 不进行自动语法恢复(没有"修复"错误的尝试)
```

```text
  │         │   └─ catch ParseException             │
  │         └─ 记录错误位置和类型                   │
  │                                                │
  │  错误信息: position=10, expected=FROM,         │
  │            actual=FORM, context=parseQuery     │
  └────────────────────┬─────────────────────────┘
                       │
                       ▼
  ┌──────────────────────────────────────────────┐
  │           第二遍解析 (错误报告)                 │
  │                                                │
  │  tokenizer.reset(errorPos=10)                  │
  │  Parser.parsePrepared() (第二次)               │
  │    ├─ tokenizer 从位置 10 开始                  │
  │    ├─ 在 parseQuery() 的 read("FROM") 处       │
  │    │   再次捕获 ParseException                  │
  │    ├─ getExpectedTokens(10):                   │
  │    │   ┌─ 上下文: AFTER_SELECT_EXPR           │
  │    │   ├─ 期望: [FROM, INTO, WHERE]            │
  │    │   └─ 返回期望 Token 列表                  │
  │    └─ 构建 SQLException                         │
  └────────────────────┬─────────────────────────┘
                       │
                       ▼
  ┌──────────────────────────────────────────────┐
  │           用户看到的错误消息                    │
  │                                                │
  │  org.h2.jdbc.JdbcSQLSyntaxErrorException:      │
  │  Syntax error in SQL statement                  │
  │  "SELECT * FORM users"                          │
  │  Expected "FROM";                                │
  │  SQL statement:                                 │
  │  "SELECT * FORM users"                          │
  │       ↑                                        │
  │       Position 10 (第 10 个字符附近)            │
  └──────────────────────────────────────────────┘
```

  关键设计决策:
    两遍解析而不是一遍解析 + 错误恢复:
      第一遍: 正常解析, 在第一个错误处停止
      第二遍: 记录期望 Token 列表, 提供更丰富的错误上下文
      优点: 实现简单, 错误信息准确
      缺点: 对长 SQL 可能需要多一倍的解析时间 (但 SQL 通常很短)

  readIf() 的作用:
    readIf(optionalToken) 不会抛出异常
    在可选语法部分 (WHERE, GROUP BY 等) 中使用
    使解析器能优雅地跳过缺失的可选部分

### 6.10.8 优缺点

**优势：**
- 手工编写的递归下降解析器，错误信息可读性强
- 支持复杂的 SQL 语法（CTE、窗口函数、递归查询）

- Parser.java 近万行，维护成本高
- 不适用左递归文法（需要手动改写）
- 不支持语法驱动的自动解析（如 ANTLR）

### 6.10.9 设计权衡

**手工编写 vs. 解析器生成器（ANTLR/JavaCC）：**

| 特性 | 手工 Recursive Descent | ANTLR 生成 |
|------|----------------------|------------|
| 错误信息 | 精确可控 | 需要额外工作 |
| 性能 | 极优 | 中等 |
| 维护成本 | 高（每次 SQL 语法变更都改代码） | 低（改文法文件） |
| 集成度 | 完全控制 | 需要适配层 |
| 代码量 | ~1 万行 | ~300 行文法 + 监听器 |

**H2 为什么选择手工实现：**

1. H2 支持超过 50 种 SQL 语句，手工解析器每行代码都是可控逻辑
2. 解析器与 H2 的 type system、schema lookup、session 状态紧密耦合
3. 嵌入式数据库需要精确控制解析性能
4. ANTLR 生成的解析器对左递归文法需要特殊处理，SQL 语法中左递归常见（如表达式）

**两阶段设计（Tokenizer + Parser）：**

分离词法分析和语法分析，使 Parser 的处理更清晰。Tokenizer 在字符级别工作（大 switch），Parser 在 Token 级别工作，专注于"消费 token 并构建 AST"。


**图 6-104: Tokenizer/Parser 两阶段架构**

两阶段设计的职责和数据流转:

```text
  ┌─────────────────────────────────────────────────────────────┐
  │  Tokenizer (词法分析)                                       │
  │  输入: SQL 字符流                                           │
  │  处理: skipWhitespace() -> 字符分派 -> 选择扫描方法          │
  │  输出: Token 流 [SELECT, *, FROM, users, ...]               │
  └────────────────────────┬────────────────────────────────────┘
                           │
                           ▼
  ┌─────────────────────────────────────────────────────────────┐
  │  Parser (语法分析)                                          │
  │  输入: Token 流                                              │
  │  处理: parsePrepared() -> switch(firstToken) -> parseXxx()  │
  │  输出: ParsedStatement (AST + 列引用 + 表引用)               │

```

## 6.11 本章小结

查询优化器通过基于代价的连接顺序选择、索引条件评估和表达式优化，将声明式 SQL 转化为高效的执行计划。R-Tree 空间索引为多维数据提供了高效的范围查询能力。递归下降解析器以其可读性强、易于调试的特点，将 SQL 文本转化为语法树。这三个查询算法与前三篇的数据结构和存储算法一起，构成了 H2 从 SQL 解析到数据访问的完整链路。
