# H2 Database 源码分析 — 写作风格与可读性提升计划

Created: 2026-06-11

## 背景

H2 Database 源码分析文档（`docs-stm/`）已完成全部 12 章交付（v4.24），82 项 final_check 全部通过，4 轮审查问题全部关闭。当前文档的功能完整性已达到交付标准，但写作风格和可读性距离专业出版社级中文技术书籍（如机械工业出版社、电子工业出版社博文视点、人民邮电出版社图灵系列）仍有提升空间。

本计划聚焦**写作风格与可读性**单一维度，目标是在不改变章节结构和技术内容的前提下，将全书的语言质量提升至出版社级中文技术著作的水平。

---

## 总览

| 维度 | 当前状态 | 目标 |
|------|---------|------|
| 句式多样性 | 部分章节句式结构重复（如频繁以"H2"或"其"开头） | 句式变化丰富，长短句交替有节奏 |
| 用词精准度 | 动词选择有限（"分析""介绍""讨论"重复使用） | 每个语境使用最精确的动词 |
| 段落组织 | 部分段落超过 10 行，缺乏内部层次 | 段落聚焦一个论点，6-8 行内自然断句 |
| 术语一致性 | 核心术语基本一致，偶有同义混用 | 全书术语严格统一，首次出现必定义 |
| 中英混排 | check_style 已检出 2 处中英混合内联代码 | 中英文排版规范（空格、引号、标点）100% 一致 |
| 代码与正文配合 | 代码引用完整，但部分说明与代码耦合不够紧密 | 代码段前有引导、段中有标注、段后有解释 |
| 过渡衔接 | 节与节之间过渡自然度参差 | 每节开始有承上句，结束有启下句 |
| 口语化程度 | 仅 1 处口语化表达（"你会发现"） | 全文保持统一的学术性正式口吻 |

## 阶段一：风格基线与检测工具增强

为所有后续润色工作提供定量基准和自动化支撑。

### 1.1 增强 check_style.py 检测能力

当前 check_style.py 仅覆盖 8 种口语化模式、长句中文字符计数和中英混合内联代码。需扩展以下检测项：

**新增检测模式：**

| 类别 | 检测模式 | 示例 | 处理建议 |
|------|---------|------|---------|
| 句式单调 | 连续多句以同一词开头（H2／该／这个／通过） | "H2 提供了...H2 支持...H2 的..." | 被动句、主题句交替 |
| 冗余副词 | "实际上""基本上""本质上""其实" | "实际上 H2 采用 MVCC" | 直接陈述 |
| 空泛修饰 | "非常""十分""极其""相当" | "非常高效" | 改为具体描述或删除 |
| 弱动词 | "进行" + 动词 | "进行分析""进行讨论" | 直接使用动词："分析""讨论" |
| 过度"的" | "的" 在连续修饰中密集出现 | "MVStore 的基于 B-Tree 的存储架构的..." | 重新组织修饰结构 |
| 模糊指代 | "其"指代不明确 | "其实现方式..."（前句有两个候选主语） | 明确指代对象 |
| 重复连接词 | 段落内反复使用"同时""另外""此外" | "同时 H2 支持...同时还可以..." | 替换为因果/转折/递进连接词 |
| 被动滥用 | "被" 的过度使用 | "数据被写入后被读取" | 主动句优先 |

**实现方式：**
- 在 check_style.py 中新增 `# 新增检测模块 X` 区域
- 每个模式使用独立函数，便于开关和调参
- 分级输出：WARN（建议修改）/ INFO（供参考）
- 按章节汇总统计，生成章节级风格评分

### 1.2 新增可读性量化指标

在现有 `readability_check.py` 基础上扩展：

- **平均句长**：每章去除代码围栏后的中文平均句长（含标点）
- **段落密度**：正文段落平均行数分布
- **术语密度**：每百字核心术语出现频率
- **"的"密度**：每百字"的"字出现频率（中文"的"过度使用指数）
- **句式起始多样性**：统计段落中句首词的变异系数
- **引导块覆盖率**：各章引导块（本章导读/前置知识/章节要点）的完整性和一致性

### 1.3 基线报告生成

对所有 9 个源章节文件运行增强后的检测：

- 输出章节级风格报告：每项指标、检出的具体位置和全文统计
- 标记需要优先处理的章节（指标偏离均值 1.5 个标准差以上）
- 基线报告作为后续润色工作的进度参考

**产出：** `docs-stm/tools/check_style.py`（增强版）+ 章节风格基线报告

## 阶段二：逐章语言润色

按章节顺序，对 9 个源文件进行系统性语言润色。优先级跟进阶段一基线报告。

### 2.1 润色规程

每章的润色遵循固定的五步流程：

1. **通读**：完整通读一章，标注句式单调段、用词不当、衔接生硬处
2. **句级优化**：逐句检查——句式多样性、主谓宾清晰度、被动语态是否必要
3. **段级重组**：检查段落焦点是否单一、信息密度是否适中、句间过渡是否自然
4. **术语检查**：对照术语表（`back/glossary.md`），确保首次出现必定义，同义术语统一
5. **代码-正文配合**：检查每个代码围栏是否有引导句、代码中的关键行是否在正文中有注释

### 2.2 润色重点

**第7-8章（SQL 执行与优化器，8,043 行）和第9-10章（持久化与锁，6,515 行）** 为优先润色目标——这两个文件行数最多，句式重复率较高，且有大量代码-正文配合需要优化。

**第4-5章（模块与流程，3,881 行）和第6章各子章（数据结构+算法，10,897 行合计）** 为第二优先级——内容密度高，段落偏长，需要在维持信息密度的同时改善可读性。

**第1-2章（架构，2,105 行）、第3章（包结构，2,223 行）、第11-12章（总结，2,306 行）** 为第三优先级——行数较少，当前质量相对较高，以微调为主。

### 2.3 润色原则

- **忠实于内容**：不改变技术含义、源码引用和章节结构。仅优化语言表达
- **最小改动**：能改词不改句，能改句不段落，避免大段重写
- **保持作者声音**：保留原文的分析视角和行文气质，不强求改写为统一模板
- **逐章提交**：每章润色完成后独立验证（`final_check.py` 通过 + 人工通读确认）

### 2.4 验证标准

每章完成润色后需通过：
- `final_check.py` 82/82 全部通过（结构完整性不因润色而破坏）
- `check_style.py` 增强版该章警告数为下降趋势
- 润色修改不会改变图号、源码路径、交叉引用、章节标题

**产出：** 9 个源文件的润色版本

## 阶段三：全局一致性检查

在逐章润色完成后，做一次跨章节的统稿。

### 3.1 术语一致性审计

- 遍历所有章节，提取所有技术术语（类名、算法名、概念名）
- 对照 `back/glossary.md`，标记：
  - 使用了术语但未收入术语表（需加条目）
  - 术语有多种中文译法（需统一为标准译法）
  - 术语首次出现未标记（需在脚注或括号中标注定义）

### 3.2 句式和口吻一致性审计

- 检查各章节的"章节导读"格式是否完全一致
- 检查"本章小结"的写作口吻是否一致（不应有的用结论式，有的用列举式）
- 检查延展阅读的格式和深度是否一致

### 3.3 引导和总结体系完整性

- 确保每章的引导块都包含：本章导读、前置知识、章节要点、术语参考四项
- 确保每章的小节包含实质总结（而非简单复述目录标题）
- 确保章末延展阅读包含可操作的资料链（不仅列标题，还说明阅读价值）

**产出：** 术语表增量更新 + 引导块和总结块统稿

## 阶段四：阅读体验优化

语言质量提升后，做读者体验层面的增强。

### 4.1 段落内部标点优化

- 长句拆分：超过 80 字的句子按意群拆分为 2-3 句
- 列举结构规范化：同一段落内多项列举使用一致的连接方式
- 技术层级性表达优化：多层嵌套的描述（如"A 的 B 的 C"结构）改为树状或分句表达

### 4.2 交叉引用质量提升

- 检查"详见第 X 章"引用格式的句式丰富度（不要千篇一律的"详见"）
- 对跨章引用的上下文增加一句话铺垫（让读者知道"为什么要去那里看"）

### 4.3 代码注释语言审计

- 检查书中 Java 代码片段里的注释是否为中文注释（统一为中文注释，符合全书语言设定）
- 检查源码位置标注格式的一致性（`org/h2/...` 格式 vs Java 类的完整引用）

**产出：** 阅读体验增强的改进集

## 阶段五：质量锁与维护

将提升后的质量标准化为可持续的新基线。

### 5.1 写作风格检查加入标准流水线

将增强后的 `check_style.py` 写为可选流水线的一部分，并建议在每次源文件变更后运行：

```bash
python docs-stm/tools/check_style.py
```

### 5.2 新增质量门禁（可选）

在 `final_check.py` 或 `docs-stm/management/testplan.md` 中，考虑增加：
- 写作风格警告数上限（如 WARN 级别不超过 10 条）
- 章节风格评分一致性要求（各章之间偏差不超过 2 个标准差）

### 5.3 风格指南文档

将本书的语言约定整理为一份轻量级风格指南，包含：
- 术语选用原则
- 中英混排规范
- 常用句式模板（非强制，但提供参考）
- 代码注释语言约定

这份指南可作为后续新章节写作或修改时的参考规范，也可作为 `check_style.py` 规则的说明文档。

**产出：** 风格检查常态化 + 风格指南文档（可选）

## 关键决策

| 决策 | 选项 | 建议 |
|------|------|------|
| 润色顺序 | 按章节顺序 vs 按问题严重程度 | 建议按问题严重程度（阶段一基线报告决定优先级），效率更高 |
| 阶段一和阶段二串行还是并行 | 工具增强完成后开始润色 vs 两者并行 | 工具增强和润色可以并行启动——增强检测能力的同时开始高优先级章节的润色 |
| 是否需要人工通读验证 | 仅工具检查 vs 工具+人工通读 | 建议每章润色后至少做一次人工通读。工具检查结构性问题，人工通读发现文学性问题 |
| 风格指南的详略程度 | 轻量级（2-3 页）vs 完整手册（10+ 页） | 轻量级即可，覆盖必须约定即可。过度规范会抑制写作灵活性 |

## 风险与应对

| 风险 | 应对 |
|------|------|
| 润色引入新的格式问题（图号错乱、引用断裂） | 每章润色后必须运行 final_check.py 82 项全过 |
| 过度润色导致内容失真 | 严格限定改动范围：不改技术含义，不改源码引用，不改目录结构 |
| 不同章节润色不一致（一部书两种风格） | 阶段三的统稿专门解决此问题 |
| 检测工具产生噪声（假阳性过多） | 新增检测模式默认 INFO 级别，稳定后再提升为 WARN |
