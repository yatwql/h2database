# 写作风格指南

> 轻量级风格参考，适用于 H2 源码分析文档的写作与修订。
> 版本：v6.0 · 2026-06-15
> 变更：v4.27 → v6.0 新增 §12 思考题写作 / §13 端到端案例叙事 / §14 图注动宾结构

---

## 1. 术语选用原则

### 1.1 核心术语统一

| 推荐用词 | 避免用词 | 说明 |
|---------|---------|------|
| 存储引擎 | 存储层引擎、底层存储 | 全书统一为"存储引擎" |
| 查询优化器 | 优化器、CBO | 首次出现可使用"查询优化器（CBO）" |
| 持久化 | 持久性、存储 | 与 ACID 的"持久性"区分时用"持久化机制" |
| 压缩整理 | 压缩、Compaction | 指 MVStore 的空间回收流程 |
| 快照隔离 | Snapshot Isolation | 首次出现标注英文全称 |
| 写偏斜 | 写倾斜、Write Skew | 统一为"写偏斜" |

### 1.2 中英文混排

- **首次出现的技术术语**：中文名（英文名），如"写放大（Write Amplification）"
- **已定义术语**：后续使用中文名，无需重复英文
- **类名/方法名**：使用代码反引号，如 `MVMap.get()`，不翻译
- **文件路径**：使用 `org/h2/...` 格式（非 `h2/src/main/java/org/h2/...`）

### 1.3 缩写规范

- **首次出现**：全称（缩写），如"Copy-on-Write（COW）"
- **后续**：使用缩写
- **避免**：在同一段落中频繁切换全称和缩写

## 2. 句式风格

### 2.1 推荐句式

| 场景 | 推荐 | 避免 |
|------|------|------|
| 描述目的 | "X 用于实现 Y" | "X 是用于实现 Y 的" |
| 引用章节 | "详见第X章《标题》"（多样性：可替换为"参见""参考""见"） | 全书千篇一律"详见" |
| 总结观点 | "综上所述，X..." | "总的来说，X..." |
| 引入概念 | "X 是一种 Y，其核心思想是..." | "X 本质上是一种 Y" |

### 2.2 句首多样性

避免连续 3 句以上以同一词开头。参见 `check_style.py` 的句式单调检测：

```text
连续以"H2"开头 → 交替为"该引擎""数据库""系统"
连续以"该"开头 → 交替为"此""这种""上述"
连续以"其"开头 → 交替为"该组件的""该机制的"或具体名词
```

### 2.3 动词选择

| 弱动词 | 强动词 |
|--------|--------|
| 进行分析 | 分析 |
| 进行讨论 | 讨论 |
| 进行优化 | 优化 |
| 是非常高效的 | 效率较高 / 延迟较低 |
| 被用来实现 | 实现 |

## 3. 段落组织

### 3.1 段落长度

- 正文段落：6-10 行为宜，不超过 15 行
- 每个段落聚焦一个论点，段落第一句为主题句

### 3.2 列举结构

- 短列举（2-3 项）：使用顿号或括号内嵌
- 长列举（4 项以上）：使用项目符号列表或编号列表
- 同一段落内多项列举使用一致连接方式

### 3.3 技术层级表达

避免"的"字嵌套（A 的 B 的 C）：

```text
避免：MVStore 的基于 B-Tree 的存储架构的扩展性
推荐：MVStore 基于 B-Tree 的存储架构，其扩展性...
```

---

## 4. 代码注释语言约定

### 4.1 代码围栏中的注释

- Java 代码片段中的注释使用**中文注释**
- 注释内容应说明"为什么"而非"是什么"

### 4.2 源码位置标注

- 使用 `org/h2/...` 路径格式
- 类名反引号标注：`MVStore.java`
- 方法名括号标注：`get()`，`put(key, value)`

---

## 5. 交叉引用

### 5.1 章节引用格式

- 标准格式：`详见第X章《章节标题》`
- 多样化：可替换为"参见""参考""见""在...中详细讨论"
- 引用同一章内的小节：`见第X.X节` 或 `详见§X.X`

### 5.2 跨章引用铺垫

需要引用其他章节时，在引用前添加上下文说明：

```text
推荐：B-Tree 的插入过程涉及节点分裂（详见第6章第6.1.4节），其核心逻辑...
不推荐：详见第6章第6.1.4节。B-Tree 的插入过程...
```

---

## 6. 排版规范

### 6.1 中英文混排

- 中文与英文之间保留空格：`H2 数据库`（不是 `H2数据库`）
- 中文与数字之间保留空格：`第 2 章`（不是 `第2章`）
- 引号使用中文引号：`"X"`（英文引号仅用于代码引用）

### 6.2 图注格式

- 格式：`**图 X-Y: 标题**`
- 独占一行，位于代码围栏或表格外部
- 引用时使用"如图 X-Y 所示"

### 6.3 列表格式

- 无序列表：`- `（减号+空格）
- 有序列表：`1. `（数字+句点+空格）
- 子列表缩进：2 个空格

## 7. 引导块结构

每章标配四个元素，严格按此顺序：

```text
> **本章导读**: ...
> **前置知识**: ...
> **章节要点**: ...
> **术语参考**: ...
```

双章文件（如 ch1-2）两章各有独立的引导块。

## 8. 常见反例对照

以下为写作风格增强过程中发现的实际反例，按类别组织。

### 8.1 被动滥用

```text
不推荐：新的 Page 被添加到 chunk 末尾后被写入磁盘。
推荐：新 Page 添加到 chunk 末尾后写入磁盘。
```

```text
不推荐：该文件被存储在 FileStore 中。
推荐：该文件存储于 FileStore 中。
```

### 8.2 句式单调

```text
不推荐（连续"第X步"）：
  第一步解析 SQL，第二步生成计划，第三步执行，第四步返回结果。
推荐（动词多样化）：
  先从 SQL 解析开始，接着生成执行计划，然后执行并返回结果。
```

```text
不推荐（连续"X 节介绍"）：
  7.1 节介绍总架构，7.2 节介绍编译过程，7.3 节介绍优化器。
推荐（省略"节"）：
  7.1 介绍总架构，7.2 分析编译过程，7.3 深入优化器实现。
```

### 8.3 空泛修饰

```text
不推荐：该算法非常高效，性能十分出色。
推荐：该算法时间复杂度为 O(log n)，在 100 万条记录下查询延迟 < 1ms。
```

```text
不推荐：这是一个相当复杂的机制。
推荐：该机制涉及 5 个组件的协同：Parser、Optimizer、Executor、Transaction、MVStore。
```

### 8.4 冗余副词

```text
不推荐：H2 实际上使用了 MVStore 存储引擎。
推荐：H2 使用 MVStore 存储引擎。
```

```text
不推荐：这本质上是一种乐观锁机制。
推荐：这是一种乐观锁机制。
```

### 8.5 模糊指代

```text
不推荐：Session A 持有锁，其他需要等待。
推荐：Session A 持有锁，其他 Session 等待获取该锁。
```

```text
不推荐：B-Tree 与其 Page 节点，其分裂过程涉及其重新分配。
推荐：B-Tree 及其 Page 节点，该分裂过程涉及键值的重新分配。
```

### 8.6 过度"的"密度

```text
不推荐：MVStore 的基于 B-Tree 的存储架构的扩展性较好。
推荐：MVStore 基于 B-Tree 的存储架构具有较好的扩展性。
```

### 8.7 弱动词构造

```text
不推荐：对数据进行分析后进行讨论。
推荐：分析数据后进行讨论。
```

## 9. 自动化检查注意事项

### 9.1 check_style.py 已知限制

| 检查项 | 易误报场景 | 人工判断建议 |
|--------|-----------|-------------|
| 句式单调 | 代码块中的文件名（`Tokenizer.java`）或枚举步骤被识别为连续句首 | 检查是否在代码围栏外，中文句首重复是否合理 |
| 重复连接词 | "同时"既作连词又作时间副词（"同时执行 A 和 B，同时监控 C"） | 判断"同时"是否在同一语境下使用 |
| 被动滥用 | "被"在技术描述中有时更自然（"被序列化为字节流"） | 确认是否可改为主动句而不损失准确性 |
| 空泛修饰 | 引述官方文档评价时保留原措辞 | 保留引述内容，仅在自行描述时替换 |
| 模糊指代"其" | 前文虽多名词但指代关系清晰 | 确认无歧义即可 |

### 9.2 final_check.py 检查项

| 检查 | 说明 |
|------|------|
| 图号连续性 | 验证每章图号无跳号、无重复（含字母后缀如 6-72b） |
| 交叉引用 | 验证"详见第X章"的目标章节存在且非自身 |
| 围栏平衡 | 所有 \`\`\` 成对出现，无奇数围栏；HTML `<pre><code>` 平衡 |
| UTF-8 编码 | Markdown 和 HTML 文件均为有效 UTF-8 |
| 标题层级 | H1 → H2 → H3 → H4 无层级跳跃 |
| HTML TOC | 所有章节标题出现在 TOC 中，锚点一致无断链 |
| 合并文档 | 源文件总行数与合并文档一致 |
| CSS 样式完整性 | 生成 HTML 的 CSS 特性完整 |
| 术语/索引工具 | build_glossary.py 和 build_index.py 语法正确 |
| 索引完整性 | 索引 ≥ 120 条；每章 ≥ 5 条；章节引用有效 |
| 术语完整性 | 术语 ≥ 60 条；每条有章节引用；引用有效 |
| 写作风格 | 调用 check_style.py 输出零警告（建议性） |

### 9.3 报告解读

```text
WARN 级别 -> 建议修改，影响阅读质量
INFO 级别 -> 建议参考，可根据上下文判断是否保留
```

## 11. 段落与过渡

### 11.1 段落长度

- **正文段落**：6-10 行为宜，不超过 15 行（代码围栏和列表不计入）
- **段落聚焦**：每个段落聚焦一个论点，段落第一句应为主题句
- **拆分策略**：超长段落（超过 15 行）应在逻辑转折处拆分为两个段落。常见拆分点包括：
  - 引入新概念或子概念时
  - 从原理描述转向示例说明时
  - 从正面论述转向局限性讨论时
- **避免单行段落**：正文段落不宜为单行（章节标题、图注、列表项等不受此限）

### 11.2 章节过渡原则

每章结束时应有明确的过渡语句指向下一章，遵循以下原则：

- **位置**：位于"本章小结"的末尾或"延展阅读"的开头
- **句首模式**：
  - "本章对 X 的分析为后续第 N 章《...》讨论 Y 奠定了基础。"
  - "以上 X 和 Y 构成了第 N 章《...》深入分析 Z 的前提。"
  - "第 N 章将在本章基础上，进一步展示..."
- **具体性**：必须具体说明下一章的主题（提及章节号和标题），避免泛泛的"后续章节"
- **关联性**：过渡语句应明确展示两章内容的逻辑递进关系

### 11.3 章内递进模式

同一章内的大节，在"本章概述"或节开头处提供上下文递进说明：

```text
推荐：前两节建立了 B-Tree 的结构模型，本节转入写入路径分析。
不推荐：本节讨论写入路径。
```

### 11.4 渐进式展开（Progressive Disclosure）

技术概念应遵循"是什么 → 为什么 → 怎么用"的三层递进：

| 层级 | 核心问题 | 典型句式 |
|------|---------|---------|
| What | 是什么 | "X 是一种 Y，用于..." 或 "X 的核心思想是..." |
| Why  | 为什么需要 | "X 解决了 Y 问题" 或 "X 的优势在于..." |
| How  | 具体实现 | "X 的实现分为三步：首先...然后...最后..." |

- 对于单节内的核心概念，三层递进应在一段内连续完成
- 对于跨节的复杂概念，前节完成 What+Why，后节展开 How
- 避免跳过 Why 直接进入 How（读者无法理解设计动机）

### 11.5 示例完整性

- 每个核心算法和流程至少有一个完整的代码示例（` ```java ` 围栏），包括完整的类声明、方法签名和逻辑注释
- 代码示例应展示"使用场景 → 核心逻辑 → 预期输出"的完整链路
- 避免纯伪代码片段而无上下文的用法

## 12. 思考题写作

每章末尾设 `## N.X 延伸思考` 小节，3-5 题。思考题不是判断对错的考题，而是引导读者将本章知识转化为可应用思维的工具。

### 12.1 题型分类

按主流技术教材的成熟分类，使用三种题型：

| 题型 | 标记 | 目的 | 占比 |
|------|------|------|------|
| **理解** | 🟢 | 检验读者能否复述本章关键概念、机制或决策 | 约 40% |
| **分析** | 🔵 | 引导读者推断"如果 X 改变，Y 会怎样"或对比两种方案 | 约 40% |
| **动手** | 🟠 | 鼓励读者动手验证、阅读源码、跑测试或修改代码 | 约 20% |

### 12.2 难度标记

每题前置星级，便于读者按目标筛选：

- **★** — 仅需复述本章内容，无外部知识
- **★★** — 需要综合本章与前置章节，或推理 1-2 步
- **★★★** — 需要阅读源码、跑实验或跨章联想

### 12.3 题目结构（强制三要素）

每题必须包含三要素：题干 + 提示 + 回顾。

**模板**：

```markdown
**1. 🟢★ 为什么 MVStore 选择 Append-Only 写入而非原地更新？**

> 提示：考虑顺序写 vs 随机写的 I/O 代价、崩溃恢复的简化、以及 Copy-on-Write 与版本管理的天然契合。
> 回顾：§9.1 MVStore 架构、§6.2 Copy-on-Write
```

要素含义：

- **题干一行**：以问句、动词起始的祈使句或对比性陈述呈现，避免模糊词如"讨论一下"。
- **提示一行**：用 `>` 引用块，给出 1-2 个思考角度，不直接给答案。提示行长度 30-60 字。
- **回顾一行**：用 `>` 引用块，列出 1-3 个章节锚点（`§X.Y` 或 `§X` 或 `第X章`），读者可直接回查。

### 12.4 题型示例

**理解题（🟢★）**：

```markdown
**1. 🟢★ 描述 B-Tree 节点分裂触发的条件，以及分裂后父节点如何更新。**

> 提示：关注节点容量阈值与父节点子指针数组的递归更新。
> 回顾：§6.1.3 节点分裂
```

**分析题（🔵★★）**：

```markdown
**2. 🔵★★ 如果将 LIRS 缓存替换策略换成 LRU，扫描型查询的命中率会如何变化？为什么？**

> 提示：LIRS 通过两个队列区分"近期重用"与"一次性访问"，LRU 没有此区分。
> 回顾：§6.5 LIRS 缓存替换、§6.5.4 扫描抗性
```

**动手题（🟠★★★）**：

```markdown
**3. 🟠★★★ 用 H2 的测试框架编写一个 MVStore 的崩溃恢复测试：写入 100 条数据后用 `kill -9` 模拟崩溃，重启后验证数据完整性。**

> 提示：参考 `org/h2/test/store/TestMVStore.java` 中的 `testRecover()` 方法；崩溃模拟可使用 `MVStoreTool.simulateCrash()`。
> 回顾：§9.7 恢复机制、§11.3 测试框架
```

### 12.5 反例与禁忌

| 反例 | 问题 |
|------|------|
| "讨论一下 MVCC。" | 题干模糊，无具体触发点 |
| "MVCC 的优点是什么？请尽可能多地列举。" | 缺乏聚焦，难以判断回答深度 |
| "为什么 H2 是好的数据库？" | 主观、无明确范围、与本书技术深度不匹配 |
| 题干 + 答案直接给出 | 失去思考价值；改用提示行引导思路 |
| 缺少回顾链接 | 读者卡壳时无回查路径 |

### 12.6 章节集成

- 位于章末"本章小结"之后、"延展阅读"之前。
- 双章文件（如 `ch1-2`、`ch4-5`）每个独立章末各设一组延伸思考。
- 不设独立答案附录；提示行+回顾链接已足够引导。

## 13. 端到端案例叙事

附录 A《端到端案例研究》采用"叙事 + 回指"模式：以一个用户视角的具体场景为主线，串联各章节关键步骤；不复制正文段落，仅以 `(详见 §X.Y)` 形式回指原章节。

### 13.1 案例结构（五段式）

每个端到端案例使用统一五段式结构：

1. **问题陈述**：1 段，从用户视角描述触发场景。"开发者执行 `SELECT * FROM users WHERE id = 42`，过了 8 毫秒数据返回。这 8 毫秒里，H2 内部发生了什么？"
2. **流水线骨架**：1 张全链路 ASCII 序列图，标注每个步骤的"触发组件 → 数据结构 → 决策点"。
3. **逐步剖析**：6-8 个段落，每段对应骨架中的一步，含动机、组件、关键源码片段（保留 `.java:行号`）、决策点剖析、`(详见 §X.Y)` 回指。
4. **关键决策回顾**：1 张表，列出本案例覆盖的 3-5 个核心设计决策（如"为何选 Append-Only"、"何时触发 commit"），每行含决策、原因、回指。
5. **思考小结**：3 个引导读者自查的问题，与各章末延伸思考风格一致。

### 13.2 叙事开篇句式（可选三种）

避免千篇一律的"本案例展示...":

- **场景式**："凌晨 3 点，某线上服务在执行...",
- **问题式**："为什么 H2 在崩溃后能恢复出一致状态？这一节用一次具体的崩溃事故串联...",
- **对比式**："同样是写入一行数据，单机模式与 MVStore 持久化模式经历的路径完全不同。本案例并列展示两条链路..."

### 13.3 回指规范

回指承担"跳转回原章节"的功能，必须满足：

- **格式**：`(详见 §X.Y)` 或 `(详见 §X.Y《章节标题》)`，统一用半角括号、半角冒号
- **粒度**：精确到 `§X.Y` 三级标题；只有当全章相关时才用 `(详见第X章)`
- **密度**：每段 1-2 处回指，不超过 3 处（密集回指降低叙事流畅度）
- **禁止**：不得在回指处复制原章节段落超过 1 句话；超过 1 句视为双源，应删除并仅保留回指

### 13.4 200 字示例

```markdown
**步骤 3：Parser 把 SQL 文本变成 AST**

执行链进入 `Parser.parseSelect()` 后，递归下降解析器逐 token 推进。
此时 H2 已为 `users` 表预编译过的 `Schema` 对象提供给 Parser 复用，
解析器无需查询元数据即可绑定列引用。Parser 输出一棵以 `Select` 为根的
表达式树，其中 `WHERE id = 42` 被还原为 `Comparison(Column(id), 42)` 节点。
此处的关键决策是：H2 不构造独立的 AST 抽象层，而是直接构造可执行的
`Expression` 树——这种"AST 即执行计划"的紧耦合简化了内存模型，但代价是
表达式重写阶段必须在同一棵树上原地变换 (详见 §6.10 Parser 递归下降解析、
§8.2 执行计划)。
```

### 13.5 与正文双源避免

- 案例不重复正文中的概念定义、源码行号清单、完整流程图
- 案例只做"叙事重组"——把分散在各章的步骤按时间顺序串成一条主线
- 任何"案例独有"的内容（如比较两种崩溃场景的恢复差异），优先沉淀到正文相应章节，附录中只回指
- 维护期：当正文章节号变更时，附录所有 `(详见 §X.Y)` 必须同步更新；由 `final_check.py` 校验

### 13.6 图簇桥接叙事

工业级技术书在三张及以上图集中出现时，会用一句话**桥接叙事**点明这组图的协调关系——读者不应被一连串图轰炸得失去主线。本规范将"3 张及以上图在 ≤ 40 行内连续出现"定义为一个**图簇**（figure cluster），由 `_audit_figure_clusters.py` 自动识别。

#### 桥接句式（任选其一）

- **共同视角式**：「以下 N 张图共同呈现 *主题* 的 N 个视角：*侧面1*、*侧面2*、*侧面3*。」
- **维度并列式**：「下面 N 张图分别从 *维度1*、*维度2*、*维度3* 三个角度刻画 *主题*。」
- **层次拆解式**：「这组图把 *主题* 拆成 N 个层次：图 X-Y 描绘 …，图 X-Y' 展示 …，图 X-Y'' 标注 …。」
- **顺序串联式**：「下面 N 张图按 *顺序/层次* 串联：先 …，再 …，最后 …。」

#### 桥接位置

- 桥接句紧贴**首图标题之上**，前后各一空行
- 桥接句必须**点名簇内每张图**（`图 X-Y`），让协调关系显式
- 簇内不再插入第二条桥接句——一个簇一根线

#### 反例（应改写）

- 簇内三张图前没有任何叙事连接 → 加桥接
- 桥接句只点名首图，其余两图悬空 → 改写为完整点名
- 桥接句长度超过 3 句话（实际是把正文段落塞进桥接） → 拆分到正文，桥接保持 1-2 句

#### 检查工具

`_audit_figure_clusters.py --window 40 --needs-bridge` 输出待桥接的图簇清单。

## 14. 图注动宾结构

图注（`**图 X-Y: Title**`）是图的"一句话摘要"，应让读者**不看图本体也能猜出图在讲什么**。工业级技术书图注采用动宾结构：动词 + 宾语 + 限定语三段式。

### 14.1 三段式模板

```text
**图 X-Y: <动词短语> <宾语短语> <限定/语境短语>**
```

- **动词短语**：展示 / 对比 / 拆解 / 演示 / 标注 / 概览 / 追踪
- **宾语短语**：图的主体内容（具体的对象/过程/关系）
- **限定/语境短语**：可选，限定时机/条件/视角

### 14.2 正例 5 条

| 正例 | 三段式拆解 |
|------|-----------|
| 展示 B-Tree 节点分裂时父节点的递归更新 | 展示 / B-Tree 节点分裂时父节点的递归更新 / — |
| 对比 LRU 与 LIRS 在扫描型负载下的命中率曲线 | 对比 / LRU 与 LIRS 的命中率曲线 / 在扫描型负载下 |
| 追踪一条 SELECT 从 JDBC 到 FileStore 的调用链 | 追踪 / 一条 SELECT 的调用链 / 从 JDBC 到 FileStore |
| 演示 Copy-on-Write 写入时新旧版本节点的并存 | 演示 / 新旧版本节点的并存 / 在 Copy-on-Write 写入时 |
| 拆解 Chunk 文件格式：Header、Page 区、Footer 三部分 | 拆解 / Chunk 文件格式 / Header、Page 区、Footer 三部分 |

### 14.3 反例 5 条

| 反例 | 问题 | 改写 |
|------|------|------|
| **节点分裂** | 仅名词，缺动词与限定 | 展示 B-Tree 节点分裂触发条件与父节点更新 |
| **过程图** | 抽象到无意义 | 拆解 SELECT 编译过程的三个阶段 |
| **MVStore** | 缺主语动作 | 概览 MVStore 的四层组件结构 |
| **示例** | 完全无信息 | 演示 INSERT 语句的完整执行路径 |
| **架构** | 模糊词 | 标注 H2 接入层与引擎层的边界 |

### 14.4 长度边界

- **最短**：8 个汉字（动词+宾语已足够明确）
- **最长**：30 个汉字（超过 30 字应拆为图注+正文叙述）
- **中位**：15-20 个汉字

少于 8 字的图注通常缺少限定语境，建议补充；超过 30 字的图注通常把正文段落塞进图注，建议拆分。

### 14.5 检查工具

`_audit_captions.py`（在 v5.5 / Phase E 引入）按本节规范扫描全书图注，输出违规清单：

- 仅名词短语（无动词）
- 长度 < 8 字或 > 30 字
- 含模糊词（"过程"、"示例"、"架构"等且无具体限定）

修复时图号保持不变，仅修改 Title 文本。
